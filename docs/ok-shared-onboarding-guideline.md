# Onboarding Guideline: ok-shared Cluster — Provisioning and Central Grafana

> **Language convention:** This is the authoritative EN version for the repository (`docs/ok-shared-onboarding-guideline.md`). A German translation is published to Confluence. In case of discrepancy, this version governs.

**Scope:** This document defines the implementation guardrails for provisioning the `ok-shared` cluster and deploying Central Grafana as the first shared platform service. It makes no new architectural decisions — those remain in **ADR-Platform-020 (Shared Platform Services)** and **ADR-Platform-018 (Observability Capability)**. This guideline lives one level below: Implementation Profile and Provider Values.

**Audience:** Engineers onboarding to ok-shared without having been part of the ADR-020 review.

**Source of truth for tasks:** Jira ticket OK-81. This guideline does not duplicate its task list or acceptance criteria — it adds the guardrails and platform-local knowledge the ticket assumes.

> **Compliance note:** ok-shared is an internal platform cluster. Do not expose services externally without an explicit ingress decision per ADR-010.

---

## Part A — What has already been decided (non-negotiable)

These decisions come from ADR-Platform-020 (Status: *Accepted*, 2026-07-15) and ADR-Platform-018 (Status: *Accepted*). They are contract-level decisions. You implement within these guardrails; you do not revisit them. If anything in your implementation path collides with one of these points, stop and read Part C.

1. **ok-shared serves clusters; ok-mgmt controls them.** ok-mgmt owns Crossplane, CAPI, and XRDs — it provisions and manages cluster lifecycle. ok-shared hosts services consumed by all registered clusters: identity, storage, registry, and observability federation. Never deploy shared services on ok-mgmt and never run cluster control logic on ok-shared.

2. **Central Grafana is strictly additive.** Every cluster keeps its own per-cluster observability stack and all five ADR-018 guarantees. Central Grafana adds a global view over per-cluster Prometheus via Thanos — it does not replace anything. Absence or unavailability of ok-shared must leave all per-cluster stacks fully functional. Do not remove or disable any per-cluster Grafana instance.

3. **All ok-shared integration hooks are opt-in per implementation profile.** Thanos sidecar/agent and log forwarding are never default-on modifications on the per-cluster stack. They are explicitly configured per profile. For this onboarding, you are setting up the receiving end (Central Grafana + Thanos Query) on ok-shared — the per-cluster sidecar configuration is a separate follow-up.

4. **ok-shared is registered as a workload cluster under ADR-013.** It follows the same registration contract as all other clusters: one cluster, one name, one credential source. The `ok-shared-kubeconfig` secret lives in `crossplane-system` on ok-mgmt with idempotent replace semantics.

5. **The v1 service set is fixed by the first real consumer.** For this onboarding the first service is Central Grafana (with Thanos Query as its datasource). The full v1 scope (Keycloak, MinIO, Harbor, Secret Backend, Thanos, Central Grafana, Central OpenSearch/Loki) is defined in ADR-020 — but you implement only what is needed now. Do not pre-install services that have no consumer yet.

## Part B — The implementation path

Execute in this order: **provision the cluster first**, then **bootstrap**, then **register**, then **install storage**, then **deploy Central Grafana**. Grafana without a running cluster is not a useful starting point.

### Platform-local context (Provider Values — freely adjustable)

| Item | Value | Note |
|---|---|---|
| Cluster name | `ok-shared` | Fixed by ADR-020 naming decision |
| Host node | `ok-infra` | CPU-only node, sufficient headroom |
| API Server IP | `192.168.100.208` | Verify against current MetalLB allocations — see pitfalls |
| Ingress IP | assigned by MetalLB automatically | MetalLB finds the next free IP; no manual configuration needed |
| vSwitch | VLAN 4000, `192.168.100.0/24` | Active vSwitch |
| KUBECONFIG (ok-infra) | `~/.kube/ok-infra.yaml` | Use `export KUBECONFIG=~/.kube/ok-infra.yaml` or the `okb` alias (note: `okb` exports KUBECONFIG AND runs `kubectl get nodes` — do not pipe arguments into it) |
| KUBECONFIG (ok-shared) | `~/.kube/ok-shared.yaml` | Generated by `make bootstrap`; set explicitly after bootstrapping |
| Cluster type | `TYPE=talos` (interim) | `TYPE=shared` is defined at acceptance per ADR-020 §5 — not yet implemented in ok-cluster |
| Topology | 1 CP + 3 Workers, HA=false | Sufficient for v1 services; HA is a follow-up decision |
| Storage class | `local-path` | ok-cluster installs local-path-provisioner on Talos clusters via `make install-storage`; Longhorn is not used on ok-shared in v1 |
| Thanos endpoint (future) | per-cluster Prometheus sidecars → ok-shared Thanos Query | Sidecar configuration on per-cluster stacks is out of scope for this onboarding |

### Step 1 — Provision the cluster

**Known issue (OK-82, OK-83):** `make new` does not honour `NODE=ok-infra` and assigns the next IP from an internal counter without checking existing MetalLB allocations. Both must be patched manually before bootstrapping.

```bash
export KUBECONFIG=~/.kube/ok-infra.yaml
cd ~/temp/kubernauts/ok/ok-cluster

make new CLUSTER=ok-shared TYPE=talos HA=false WORKERS=3
```

**Before bootstrapping — mandatory manual patch:**

```bash
# 1. Check current MetalLB allocations
kubectl get svc -A | grep LoadBalancer

# 2. Patch nodeSelector (make new defaults to ok-gpu)
sed -i '' 's/nodeSelector: ok-gpu/nodeSelector: ok-infra/' ok-shared/cluster-config.yaml

# 3. Patch endpoint IP to the next free IP (verify from step 1)
sed -i '' 's/endpoint: 192.168.100.203/endpoint: 192.168.100.208/' ok-shared/cluster-config.yaml

# 4. Re-render manifests with corrected values
make render CLUSTER=ok-shared

# 5. Verify
grep -E "endpoint|nodeSelector" ok-shared/cluster-config.yaml
# expected: endpoint: 192.168.100.208 / nodeSelector: ok-infra
```

Then bootstrap:

```bash
make bootstrap CLUSTER=ok-shared
# takes ~5 minutes; watches DataVolume imports and waits for all nodes Ready
```

### Step 2 — Install storage

```bash
export KUBECONFIG=~/.kube/ok-shared.yaml
make install-storage CLUSTER=ok-shared
# installs local-path-provisioner and sets it as default StorageClass
```

### Step 3 — Install ingress

```bash
make install-ingress CLUSTER=ok-shared
# installs Traefik as ok-ingress; MetalLB on ok-infra assigns the ingress IP automatically
```

Note the ingress IP from the output — you will need it for DNS entries later.

### Step 4 — Register ok-shared on ok-mgmt (ADR-013)

```bash
kubectl --kubeconfig ~/.kube/ok-mgmt.yaml create secret generic ok-shared-kubeconfig \
  --from-file=kubeconfig=$HOME/.kube/ok-shared.yaml \
  --namespace crossplane-system \
  --dry-run=client -o yaml | kubectl --kubeconfig ~/.kube/ok-mgmt.yaml apply -f -
```

Idempotent replace semantics: re-run safely if the secret already exists.

### Step 5 — Deploy Central Grafana

Central Grafana on ok-shared is a standalone Grafana instance. It provides the global cross-cluster view via Thanos Query as its datasource — that datasource is configured in a follow-up once per-cluster Prometheus sidecars are deployed.

```bash
export KUBECONFIG=~/.kube/ok-shared.yaml

helm repo add grafana https://grafana.github.io/helm-charts
helm repo update

helm install central-grafana grafana/grafana \
  --namespace monitoring \
  --create-namespace \
  --set persistence.enabled=true \
  --set persistence.storageClassName=local-path \
  --set 'adminPassword=<your-password>' \
  --set service.type=ClusterIP
```

> **zsh pitfall:** Use single quotes around `adminPassword` if the password contains `!` — zsh interprets `!` in double quotes as history expansion and will fail with `zsh: no such file or directory`.

> **Chart deprecation warning:** The `grafana/grafana` chart may print `WARNING: This chart is deprecated`. This is a known upstream warning and does not affect functionality. Note it and move on.

**Verify:**

```bash
kubectl get pods -n monitoring
kubectl get pvc -n monitoring
# PVC must be Bound before Grafana is functional
```

**Access (port-forward for now; ingress per ADR-010 is a follow-up):**

```bash
export POD_NAME=$(kubectl get pods --namespace monitoring \
  -l "app.kubernetes.io/name=grafana,app.kubernetes.io/instance=central-grafana" \
  -o jsonpath="{.items[0].metadata.name}")
kubectl --namespace monitoring port-forward $POD_NAME 3000
# open http://localhost:3000 — login: admin / <your-password>
```

Grafana will start without any datasource configured — that is expected. The Thanos Query endpoint is a follow-up.

### Teardown (for re-provisioning or testing)

```bash
# 1. Delete PVCs inside the cluster
kubectl --kubeconfig ~/.kube/ok-shared.yaml delete pvc --all -n monitoring

# 2. Delete the cluster
make delete CLUSTER=ok-shared

# 3. Clean up local kubeconfig and ok-mgmt registration
rm ~/.kube/ok-shared.yaml
kubectl --kubeconfig ~/.kube/ok-mgmt.yaml delete secret ok-shared-kubeconfig -n crossplane-system
```

### Known pitfalls

- **nodeSelector defaults to ok-gpu (OK-82):** `make new` ignores `NODE=ok-infra` — always patch `cluster-config.yaml` and re-run `make render` before bootstrapping. The mandatory patch is in Step 1 above.
- **IP collision (OK-83):** `make new` assigns IPs from an internal counter without checking MetalLB. Always verify current allocations with `kubectl get svc -A | grep LoadBalancer` before bootstrapping. The ingress IP is assigned correctly by MetalLB automatically — only the API server endpoint needs manual verification.
- **zsh history expansion:** Single-quote the `adminPassword` value if it contains `!`. Double quotes cause `zsh: no such file or directory`.
- **Grafana chart deprecation warning:** Expected, non-blocking. Note it and move on.
- **Traefik EXTERNAL-IP `<pending>`:** Expected inside the Talos cluster — traffic reaches Traefik via the RKE2-side `ok-shared-ingress` MetalLB service, not via a cluster-internal LoadBalancer.
- **KUBECONFIG context:** ok-cluster generates `~/.kube/ok-shared.yaml` during bootstrap. Set `export KUBECONFIG=~/.kube/ok-shared.yaml` explicitly after bootstrapping — the shell aliases (`okb`, `okm`, `okt`) target other clusters and will misdirect commands if used.

## Part C — The stop rule (an escalation decision rule)

This is not a prohibition — it is decision logic: it tells you *which* of two paths applies for the change at hand. The boundary between "just do it" and "escalate first" is the contract line from ADR-001: **Provider Values are yours; Contracts are not.**

**Freely adjustable (Provider Values / implementation details):**

- IPs, node assignments, namespace names, replica counts, resource requests/limits
- Helm chart values, storage class names, retention settings
- Grafana dashboard configuration, datasource URLs, alert receiver endpoints
- Anything the Jira ticket marks as a task

**Stop and escalate (contract touch → ADR amendment + three-way review):**

- **Removing or disabling a per-cluster Grafana instance** in favour of ok-shared — this is an ADR-018 amendment under ADR-017 envelope rules; ADR-020 §4 explicitly states this ADR grants no authority to weaken existing guarantees
- **Making any ok-shared hook default-on** on the per-cluster stack (Thanos sidecar, log forwarder, central OIDC) — opt-in per profile is a contract-level decision
- **Adding a new v1 service** not in the ADR-020 service table — new services require the Shared Platform Services Contract to be written first (testable guarantees per the platform method)
- **Deploying services on ok-mgmt** that belong on ok-shared (identity, registry, object storage) — the control/service boundary is a decided separation
- **Exposing ok-shared services externally** without an ingress decision per ADR-010

**How to escalate:** Open the discussion with a short written problem statement referencing the relevant ADR clause, and run it through the review checklist in `docs/platform-engineering-method.md`. The platform rule applies to us the same way it applies to the workloads we build: anyone may argue; the contract changes only through a reviewed decision.

**The test:** Would your change survive replacing Grafana's Helm chart with a different chart, without anyone noticing? Then it is a Provider Value — proceed. Would the change require touching the per-cluster stack or removing an ADR-018 guarantee? Then it touches a contract — escalate.
