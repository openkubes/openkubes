# KubeVirt Cluster Provisioning via Crossplane

This directory provides a Crossplane-based self-service API for provisioning
KubeVirt-backed Kubernetes workload clusters on OpenKubes.

> 🇩🇪 [Deutsche Version](README_DE.md)

---

## Overview

```
kubectl apply -f examples/ok1.yaml
       ↓
KubeVirtClusterClaim (platform.openkubes.ai/v1alpha1)
       ↓
Crossplane Composition
       ↓
provider-kubernetes creates:
  ├── ConfigMap  (deploy-args-ok1-<hash>)
  ├── ServiceAccount + ClusterRoleBinding
  └── Job  (deploy-ok1-<hash>)  ← runs capi-platform-runner
       ↓
deploy-full.sh → CAPI + KubeVirt → Workload Cluster
  ├── Namespace on management cluster
  ├── Namespace on infra cluster
  ├── Per-cluster infra secret (correct namespace key for LB placement)
  ├── CAPI manifests rendered + applied
  ├── Workload kubeconfig fetched
  ├── CNI installed (Calico / Cilium)
  └── Nodes Ready
```

> **Note:** The cluster name will include a short hash suffix generated by
> Crossplane (e.g. `ok1-gfkts`). This is the XR name. See Roadmap for how
> this will be resolved in a future operator.

---

## Prerequisites

- Crossplane installed on the management cluster
- `provider-kubernetes` v0.17+ with `in-cluster` ProviderConfig
- `function-patch-and-transform` v0.8+
- `kubernautslabs/capi-platform-runner:v4.2` image accessible from the cluster
- CAPI + CAPK installed on the management cluster
- Secret `external-infra-kubeconfig` present in `capk-system`

## Installation

```bash
# 0. Create the openkubes-system namespace (one-time setup)
kubectl apply -f namespace.yaml

# 1. Apply RBAC for provider-kubernetes
# First find the active ServiceAccount name:
kubectl get sa -n crossplane-system | grep provider-kubernetes
# Update the name in rbac.yaml, then:
kubectl apply -f rbac.yaml

# 2. Register the XRD
kubectl apply -f xrd.yaml

# 3. Apply the Composition
kubectl apply -f composition.yaml

# 4. Verify
kubectl get xrd kubevirtclusters.platform.openkubes.ai
kubectl get composition kubevirtcluster.platform.openkubes.ai
```

## Usage

### Deploy a cluster

```bash
# Edit examples/ok1.yaml and set your endpointIP (free MetalLB IP)
kubectl apply -f examples/ok1.yaml

# Watch the deploy Job
kubectl get jobs -n openkubes-system -w

# Follow logs
kubectl logs -n openkubes-system \
  $(kubectl get pods -n openkubes-system -l job-name \
    -o jsonpath='{.items[0].metadata.name}') -f
```

## Getting the Workload Kubeconfig

After a successful deploy, retrieve the kubeconfig of the workload cluster
using one of these two options:

### Option A – via clusterctl (recommended)

```bash
# Find the actual cluster name (XR name with hash suffix)
kubectl get cluster -A

# Example: cluster name is ok1-gfkts in namespace ok1-gfkts
clusterctl get kubeconfig ok1-gfkts \
  --namespace ok1-gfkts \
  --kubeconfig ~/.kube/<mgmt-cluster>.yaml \
  > ~/.kube/ok1-gfkts.kubeconfig

# Verify
KUBECONFIG=~/.kube/ok1-gfkts.kubeconfig kubectl get nodes
```

### Option B – via Secret

CAPI stores the kubeconfig as a Secret in the cluster namespace:

```bash
# Find the cluster name first
kubectl get cluster -A

# Extract kubeconfig from Secret
kubectl -n ok1-gfkts get secret ok1-gfkts-kubeconfig \
  -o jsonpath='{.data.value}' | base64 -d \
  > ~/.kube/ok1-gfkts.kubeconfig

# Verify
KUBECONFIG=~/.kube/ok1-gfkts.kubeconfig kubectl get nodes
```

## Delete a cluster

```bash
# 1. Delete the Crossplane Claim
kubectl delete -f examples/ok1.yaml

# 2. Delete the CAPI cluster (triggers VM deletion on infra cluster)
#    Replace ok1-gfkts with your actual cluster name
kubectl delete cluster ok1-gfkts -n ok1-gfkts

# 3. Watch VMs being removed on infra cluster
kubectl get vm -A -w

# 4. Clean up remaining namespaces on management cluster
kubectl delete ns ok1-gfkts
```

> **Note:** Automatic cleanup on `kubectl delete` is on the roadmap.
> Currently step 2 must be done manually.

## API Reference

```yaml
apiVersion: platform.openkubes.ai/v1alpha1
kind: KubeVirtClusterClaim
metadata:
  name: my-cluster
  namespace: openkubes-system
spec:
  country: de                                        # required
  provider: kubevirt                                 # required
  endpointIP: 10.10.10.50                           # required – free MetalLB IP
  cni: calico                                        # calico | cilium
  multus: "false"                                    # "true" | "false"
  controlPlane:
    replicas: 1
    kubernetesVersion: v1.34.1
  workers:
    replicas: 2
  runnerImage: kubernautslabs/capi-platform-runner:v4.2
```

## Notes

- The cluster name includes a Crossplane-generated hash (e.g. `ok1-gfkts`).
  Use `kubectl get cluster -A` to find the exact name after deploy.
- The deploy Job uses the `capi-platform-runner` ServiceAccount with
  `cluster-admin`. Scope this down to a custom ClusterRole in production.
- `compositeDeletePolicy: Foreground` ensures composed resources are deleted
  before the XR is removed.
- The runner image `kubernautslabs/capi-platform-runner:v4.2` embeds all
  platform code — no volume mount needed.
